# Research: prior art for the mental-model skill

See `borrowings.md` for the distilled mapping into skill components.

| Folder                                  | Source                                                                                                                                                                                                                                                                                                                                  | Why it matters                                                                                                                                                                                            |
| --------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `trailofbits-invariant-driven-dev`      | [Trail of Bits: The call for invariant-driven development](https://blog.trailofbits.com/2025/02/12/the-call-for-invariant-driven-development/) (2025-02)                                                                                                                                                                                | Closest named methodology. Invariant registry schema (ID / statement / components / testing strategy), Hoare-triple option, reviewers focus via registry, lifecycle: keep spec in sync on every PR.       |
| `matklad-architecture-md`               | [matklad: ARCHITECTURE.md](https://matklad.github.io/2021/02/06/ARCHITECTURE.md.html) (2021-02)                                                                                                                                                                                                                                         | Durable-doc discipline: name symbols, never link file:line; call out _absence_ invariants; document boundaries; only slow-changing facts; revisit, don't continuously sync.                               |
| `antithesis-semi-formal-proofs`         | [Antithesis: The pragmatic magic of semi-formal methods](https://antithesis.com/blog/2025/semi_formal_proofs/) (2025-05; original 404s, recovered from [Wayback 2026-02-17](http://web.archive.org/web/20260217154550/https://antithesis.com/blog/2025/semi_formal_proofs/))                                                            | Obligations pinned as code comments with review tags; touching the code obliges updating the proof + re-review. Honest cost: 2–3x. Basis for anchor comments + sync mode.                                 |
| `shardstore-lightweight-formal-methods` | [Murat Demirbas on ShardStore SOSP'21](http://muratbuffalo.blogspot.com/2021/10/using-lightweight-formal-methods-to.html) + `shardstore-sosp21.pdf` (paper)                                                                                                                                                                             | Reference models as executable specs; decompose correctness into independent properties each checked by the best-fit tool; verification ladder: sequential crash-free → sequential crashing → concurrent. |
| `kislay-distributed-review-checklist`   | [Kislay Verma: Code review checklist for distributed systems](https://kislayverma.com/programming/code-review-checklist-for-distributed-systems/)                                                                                                                                                                                       | Concrete failure-mode probes: timeouts everywhere, timeout = indeterminate not failure, idempotency, no remote calls in transactions, explicit recovery paths.                                            |
| `hillel-wayne-practical-formal-methods` | [tldrsec: PLTalk with Hillel Wayne](https://tldrsec.com/p/blog-pltalk-practical-formal-methods-hillel-wayne)                                                                                                                                                                                                                            | Target "locally correct everywhere but globally incorrect" bugs; invariants (always true) vs predicates (conditionally true); when formal-ish methods pay off and when not.                               |
| `ezyang-code-review-human-alignment`    | [ezyang: Code review as human alignment, in the era of LLMs](https://blog.ezyang.com/2025/12/code-review-as-human-alignment-in-the-era-of-llms/) (2025-12)                                                                                                                                                                              | Framing: review maintains a shared platonic model anchored on global system invariants; align the LLM with invariants _before_ review.                                                                    |
| `sridharan-mental-models`               | [Cindy Sridharan: Effective Mental Models for Code and Systems](https://copyconstruct.medium.com/effective-mental-models-for-code-and-systems-7c55918f1b3e) (recovered from [Wayback 2026-06-08](http://web.archive.org/web/20260608095209/https://copyconstruct.medium.com/effective-mental-models-for-code-and-systems-7c55918f1b3e)) | Error kernel (the part that absolutely must be correct); document implicit assumptions — "shared understanding becomes hidden knowledge".                                                                 |
